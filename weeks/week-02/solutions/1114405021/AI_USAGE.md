# AI 使用反思紀錄

## 作業背景

本作業採用 Test-Oriented Development（TDD）方法，要求「先測試，後實作」。在 Copilot 協助下完成三個任務的完整 TDD 循環。

---

## AI 協助的主要問題（3-5 條）

### 1. 如何設計全面的測試案例？
**我的問題**: Task 1 需要測試「去重、排序、提取偶數」三個操作，如何設計涵蓋正常、邊界、反例的完整測試？

**Copilot 建議**:
- 將每個操作分為獨立的測試類
- 每個類至少包含：有效性測試、邊界測試、特殊情況測試
- 邊界包括：空輸入、單元素、重複元素、已排序輸入
- 反例包括：無目標元素（如無偶數）、所有元素都符合

**採用情況**: ✅ **完全採用**  
這個建議幫我設計了高質量的測試結構。

### 2. Task 2 的排序邏輯該如何實現？
**我的問題**: 如何在 Python 中正確實現三層排序（分數降序 → 年齡升序 → 名字升序）？

**Copilot 建議**:
- 使用 `sorted(..., key=lambda x: (-x[1], x[2], x[0]))`
- 負號 `-x[1]` 實現分數降序
- 其他元素保持升序即可

**採用情況**: ✅ **完全採用**  
簡潔優雅，避免了複雜的自訂比較邏輯。

### 3. Task 3 中如何使用 Counter 和 defaultdict？
**我的問題**: Task 3 要求使用 "defaultdict 或 Counter"，兩者如何結合使用？

**Copilot 建議**:
- 使用 `defaultdict(int)` 統計用戶事件數（更直觀）
- 使用 `Counter` 統計 action 出現次數（提供 `most_common()` 方便）
- 兩者結合既滿足需求又提高代碼清晰度

**採用情況**: ✅ **完全採用**  
同時使用兩個工具，充分利用各自優勢。

### 4. 如何確保空輸入的安全性？
**我的問題**: Task 3 要求「程式需可處理空輸入（m=0）」，怎麼做最穩妥？

**Copilot 建議**:
- 在 `analyze_logs()` 函數最開始加入檢查
- 立即返回 `([], (None, 0))`
- 防止後續 `Counter.most_common()` 在空情況跳錯

**採用情況**: ✅ **完全採用**  
簡單有效，避免邊界崩潰。

### 5. 測試骨架該如何自行撰寫而非複製？
**我的問題**: 遵循 TDD 精神，測試骨架應該自行思考。如何避免模板化複製？

**Copilot 建議**:
- 先明確每個操作的「輸入/輸出規格」
- 再針對規格設計 test method（而非直譯文敘說）
- 給每個測試詳細的 docstring 說明測試目的

**採用情況**: ✅ **完全採用**  
確保每個測試都有明確的邏輯意圖，而非被動複製。

## AI 建議中的拒絕項

### 建議 1: 「使用 set 去重後轉回 list」（被拒）

**Copilot 初始建議**:
```python
def dedupe(numbers):
    return list(set(numbers))
```

**我的反應**: ❌ **拒絕理由**
- Task 1 明確禁止「不可用 set 直接輸出去重結果」
- 原因是 set 不保留順序，違反「保留第一次出現順序」的要求
- 建議雖然簡潔，但違反題目禁令

**修正**: 改用 `dict tracking` 方案，既達到去重又保留順序。

### 建議 2: 「Task 2 使用自訂比較函數」（被拒）

**Copilot 備選建議**:
```python
from functools import cmp_to_key
def compare(s1, s2):
    if s1[1] != s2[1]:  # score
        return s2[1] - s1[1]
    # ... 複雜邏輯
sorted(students, key=cmp_to_key(compare))
```

**我的反應**: ❌ **拒絕理由**
- 代碼過於複雜，容易出錯
- lambda 元組鍵方案更 Pythonic 且易維護
- Task 2 明確說「必須使用 sorted(..., key=...)」，後者更直接

**採用**: 堅持使用 `lambda x: (-x[1], x[2], x[0])` 的簡潔方案。

## AI 可能誤導但自行修正的案例

### 案例：Test 3 的 Top Action 排序

**Copilot 初始建議**:
```python
# 當有多個 action 計數相同時，返回第一個
top_action = action_counts.most_common(1)[0]
```

**問題**: 如果有多個 action 計數相同，`most_common()` 的結果依賴 Counter 內部實現（可能不穩定）。

**我的修正**:
- 檢查題目要求：「全域最常見 action 及其次數」
- 題目沒說有並列時該選哪個
- 保留 `most_common()` 的結果（已夠），但在測試中驗證計數正確性而非具體 action

**修正後的認知**: 
✅ Copilot 的建議本身沒錯，但我需要在實現中加入防禦機制和充分的測試。

## 關鍵收穫

### 1. TDD 的核心價值
通過先寫測試，我在代碼實現前就明確了邊界和需求，避免「寫出來才發現遺漏」的情況。

### 2. 充分利用 AI 的前提條件
- **明確問題**：不是「幫我寫代碼」，而是「這個需求該怎麼設計」
- **質疑建議**：遇到與題目衝突時要敢於拒絕
- **反覆驗證**：用測試驗證 AI 建議的正確性

### 3. 代碼品質的三個層次
- **Red**: 確保需求完整（測試涵蓋所有邊界）
- **Green**: 確保實現正確（所有測試通過）
- **Refactor**: 確保代碼優雅（簡潔、可讀、可維護）

---

## 致 AI 的建議

1. **多問「為什麼」**: Copilot 提供的方案都了解背後的原因嗎？
2. **不要盲目相信簡潔**: 有時一行 lambda 勝過複雜邏輯，但也要確認是否符合題意
3. **充分測試**: 代碼好看不代表隱藏的 bug 少
4. **建立習慣**: TDD 一開始覺得麻煩，但長期收益遠大於成本

本作業中 Copilot 提供的協助主要體現在「思路引導」而非「直接代碼」。透過反覆問答、測試驗證、主動質疑，最終建成了一個結構清晰、邏輯完善的 TDD 項目。
